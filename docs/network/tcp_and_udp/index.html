TCP and UDP

TCP와 UDP는 transport 계층에 존재하는 프로토콜로, Application에서 발생한 메시지(패킷)을 세그먼트로 캡슐화 하여 network 계층으로 넘기는 역할을 한다.

  • 캡슐화 (encapsulation) : 상위 계층에서 패킷을 받았을 때, 패킷에 사용될 추가 정보들을 더하는 것
    • 역캡슐화(decapsulation)도 존재한다: 하위 계층에서 패킷을 받았을 때, 하위 계층에서 추가한 정보들을 제거하는 작업
  • 패킷은 각 계층마다 불리는 이름이 다르다.
    • Application 계층: 메시지, Transport 계층: 세그먼트

TCP (Transmission Control Protocol)#

데이터의 송수신을 위해 IP를 사용하는 프로토콜 (상대방의 IP와 Port 번호를 알고있어야 연결을 요청할 수 있다.)

  • 양쪽의 소켓이 연결된 상태여야만 가능 하기 때문에 연결지향 프로토콜이라고도 부른다.
    • 소켓: 네트워크 연결을 나타내는 객체, 사용되는 소켓들은 프로토콜에 맞춰져서 구성되어야 한다.
  • 일반적으로 IP 프로토콜과 함께 쓰이기 때문에, 주로 TCP/IP 프로토콜이라고 알려져 있다.

TCP의 연결이 수립되면, 데이터를 byte stream 단위로 처리하여 전송한다.

  • 이는 데이터의 경계가 없어서, 수신부에서 데이터를 모아서 packet으로 만드는 과정이 필요하다는 의미가 된다.
  • 그리고 수신부에서 데이터의 수신 여부(ACK)를 알려주기 전까지 데이터를 재전송한다.

프로토콜 작동#

TCP는 연결에서는 3-way handshake, 연결 해제를 위해서는 4-way handshake를 사용한다.

TCP Connection (3-way handshake)#

SYN (클라이언트) -> SYN-ACK (서버) -> ACK (클라이언트) 순으로 진행된다.

  1. 먼저 open()을 실행한 클라이언트가 SYN 패킷을 보내고 SYN_SENT 상태로 대기한다. (싱크 가능 여부 확인)
    • SYN패킷에는 임의의 연속된 숫자들(Client_seq)이 포함되어 있다.
    • 패킷(packet 또는 segment): 인터넷 내에서 데이터를 보내기 위한 경로배정(라우팅)을 효율적으로 하기 위해서 데이터를 여러 개의 조각들로 나누어 전송을 하는데 이때, 이 조각을 패킷이라고 한다.
  2. 서버는 SYN_RCVD 상태로 바꾸고 SYN과 응답 ACK를 보낸다. (싱크 가능 여부 전달)
    • 서버쪽 ACK 패킷에는 클라이언트가 전송한 임의의 연속된 숫자들에 1을 더한 값(Client_seq + 1)과 서버가 만든 임의의 연속된 숫자들 (Server_seq)이 포함되어 있다.
  3. SYN과 응답 ACK을 받은 클라이언트는 ESTABLISHED 상태로 변경하고 서버에게 응답 ACK를 보낸다. (싱크 수립 후 결과 전송)
    • 클라이언트 쪽 ACK 패킷에는 서버가 전송한 임의의 연속된 숫자들에 1을 더한 값(Server_seq)과 Client_seq + 1이 포함되어 있다.
  4. 응답 ACK를 받은 서버는 ESTABLISHED 상태로 변경한다. (반대편도 싱크 수립)

TCP Disconnection (4-way handshake)#

FIN (클라이언트) -> ACK (서버) -> FIN (서버) -> ACK (클라이언트) 순으로 진행된다.

  1. 먼저 close()를 실행한 클라이언트가 FIN 패킷을 보내고 FIN_WAIT1 상태로 대기한다. (연결 종료 요청)
  2. 서버는 CLOSE_WAIT으로 바꾸고 응답 ACK를 전달한다. 동시에 해당 포트에 연결되어 있는 어플리케이션에게 close()를 요청한다. (서버에서 종료 요청을 확인함)
  3. ACK를 받은 클라이언트는 상태를 FIN_WAIT2로 변경한다.
  4. close() 요청을 받은 서버 어플리케이션은 종료 프로세스를 진행하고 FIN을 클라이언트에 보내 LAST_ACK 상태로 바꾼다. (실제 서버 종료 후 결과를 보냄)
  5. FIN을 받은 클라이언트는 ACK를 서버에 다시 전송하고 TIME_WAIT으로 상태를 바꾼다. TIME_WAIT에서 일정 시간이 지나면 CLOSED된다. ACK를 받은 서버도 포트를 CLOSED로 닫는다.

TCP 특징#

통신을 위해 연결을 수립해야 하므로, UDP 보다 속도가 느리지만 높은 신뢰성을 보장한다.

모든 TCP 연결은 전이중 (Full-duplex), 점대점(point to point) 방식이다.

  • Full Duplex: 전송이 양 방향으로 동시에 일어날 수 있음
  • Point to Point: 정확히 2개의 종단점을 가지고 있음 (클라이언트와 서버 간 1대1 연결만 가능)

흐름 제어(Flow control)과 혼잡 제어(Congestion control)을 수행한다.

  • 흐름 제어: 송신 측과 수신 측의 데이터 속도 차이를 해결하기 위한 기법
    • Stop and Wait 방식: 매번 전송한 패킷에 대해 확인 응답을 받아야만 그 다음 패킷을 전송하는 방법
  • 혼잡 제어: 송신 측의 데이터 전달과 네트워크 처리 속도 차이를 해결하기 위한 기법
    • 추가 설명: 송신 측의 데이터는 네트워크를 통해 전달되는데, 만약 특정 라우터에 데이터가 몰릴경우 혼잡이 발생할 수 있으므로, 이를 막기위해 송신 측의 데이터 전송 속도를 강제로 줄이게 된다.
    • AIMD (Additive Increase / Multiplicative Decrease) : 처음에 패킷을 하나씩 보내고, 이것이 문제없이 도착하면 window 크기 (단위 시간 내에 보내는 패킷의 수)를 1씩 증가 시켜가며 전송하는 방법. 만약, 패킷 전송을 실패하거나 timeout이 발생하면 window 크기를 절반으로 줄인다.

UDP (User Datagram Protocol)#

데이터를 데이터그램(datagram) 단위로 처리하는 프로토콜

  • 연결을 설정하지 않고 데이터를 보낼 수 있으므로, 비연결형 프로토콜이라고도 한다.
  • 데이터그램 단위라는 것은 데이터의 경계가 분명하여 수신자는 송신자가 보낸 크기와 동일하도록 강제하다는 것을 의미한다.

실시간 App (동영상 스트리밍, 화상 통화 등)은 일부 데이터 손실을 감안하고, 지연을 최소화하기 위해 UDP를 사용한다.

프로토콜 작동 (소켓 프로그래밍 절차)#

UDP는 편지와 비유할 수 있다. 데이터를 보내는 쪽에서 패킷에 데이터를 담은 후, 상대방과 자신의 IP 주소를 기록한 뒤 소켓을 통해 전송한다. 이후 패킷이 도착했는지에 대해서는 알 수 없으며, 오직 보냈다는 사실만 기억하고 있다.

데이터를 보내는 쪽은 보내는 데이터 개수만큼 sendto() 라는 소켓 API를 호출하는데, 수신하는 쪽도 받은 데이터 횟수만큼 recvfrom() 이라는 API 를 호출해줘야 한다. 이는 UDP가 데이터그램 단위로 처리하기 때문에 가능한 일이다.

note

소켓 API에 관련된 자세한 내용은 Socket 포스팅을 참조할 것.

UDP 특징#

  • 데이터 전송의 신뢰성과 순서를 보장하지 않는다.
  • 흐름 제어와 혼잡 제어를 수행하지 않고, 바로 패킷을 전송한다. 또한 연결 과정(3-handshakes)이 필요없다.
